# Version Bump Control

This project has two layers of versioning:

- source files generated by XcodeGen
- built app bundles signed by Xcode

The source of truth must be [`MacApp/project.yml`](/Users/charlie/Downloads/untitled%20folder/7zip-Mac/MacApp/project.yml), not the generated `Info.plist` or `.entitlements` files.

## Update Rules

When releasing a new version:

1. Update `CFBundleShortVersionString` and `CFBundleVersion` in both target `info.properties` blocks in [`MacApp/project.yml`](/Users/charlie/Downloads/untitled%20folder/7zip-Mac/MacApp/project.yml).
2. Do not hand-edit generated plist or entitlements files and assume they will persist after `make generate`.
3. Keep sandbox entitlements declared in both target `entitlements.properties` blocks in [`MacApp/project.yml`](/Users/charlie/Downloads/untitled%20folder/7zip-Mac/MacApp/project.yml).
4. Keep `ENABLE_APP_SANDBOX = YES` and `ENABLE_USER_SELECTED_FILES = readwrite` for both the app and the Finder extension.
5. Keep the Finder extension embed dependency on `codeSign: false` so the already-signed `.appex` is not re-signed on copy.

## Required Commands

Run these in order for every version bump:

```bash
cd "/Users/charlie/Downloads/untitled folder/7zip-Mac/MacApp"
make generate
make build
make build-release
./Scripts/create_dmg.sh "DerivedData/Build/Products/Release/SeptaZip.app" build
```

## Required Verification

Check generated source files:

```bash
plutil -p "SevenZipMac/Resources/Info.plist"
plutil -p "FinderExtension/Info.plist"
cat "SevenZipMac/Entitlements/SevenZipMac.entitlements"
cat "FinderExtension/FinderExtension.entitlements"
```

Expected:

- app version matches release version
- Finder extension version matches release version
- both entitlements files include:
  - `com.apple.security.app-sandbox`
  - `com.apple.security.files.user-selected.read-write`

Check signed build products:

```bash
codesign -d --entitlements :- "DerivedData/Build/Products/Release/SeptaZip.app" 2>/dev/null
codesign -d --entitlements :- "DerivedData/Build/Products/Release/SeptaZip.app/Contents/PlugIns/SeptaZip Finder Extension.appex" 2>/dev/null
plutil -p "DerivedData/Build/Products/Release/SeptaZip.app/Contents/Info.plist"
plutil -p "DerivedData/Build/Products/Release/SeptaZip.app/Contents/PlugIns/SeptaZip Finder Extension.appex/Contents/Info.plist"
```

Expected:

- both signed products contain sandbox entitlements
- both built products show the intended version number

Check Finder integration registration:

```bash
pluginkit -m -A -D -i com.septazip.SeptaZip.FinderExtension
```

Expected:

- the extension is listed as `com.apple.FinderSync`
- the installed path points to the app you intend to ship

## Manual QA

Before publishing any release, manually verify:

- launch the app and open a `.zip` by drag-and-drop
- `Open with 7-Zip` from Finder opens the archive without `errno=1`
- `Extract Here` works from Finder
- `Extract to Subfolder` works from Finder
- `Compress to .7z` works from Finder
- `Compress to .zip` works from Finder
- quick background jobs show progress and complete
- the Finder extension still appears after replacing the installed app

## Known Failure Modes

- If the extension disappears after a version bump, check whether XcodeGen rewrote entitlements to empty values.
- If Finder actions open the app but archive operations fail with `errno=1`, the app did not receive a valid security-scoped file grant.
- If the version reverts to `1.0`, `make generate` likely rewrote the plists from stale values in [`MacApp/project.yml`](/Users/charlie/Downloads/untitled%20folder/7zip-Mac/MacApp/project.yml).
